## Medium

### M-1 · `/api/practice/generate-station` does not check approved status
**File:** `src/app/api/practice/generate-station/route.ts:47`  
**Finding:** The route calls `getUserId(request)` and returns 401 if absent, but does not verify `user.status === 'approved'`. A `pending` user (not yet approved by an admin) can reach this route and trigger a Claude Haiku API call, incurring cost.  
**Fix:** Add an approved-status check after the user lookup:
```ts
const user = await getUser(request);
if (!user || user.status !== 'approved') {
  return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
}
```
`lib/auth.ts` does not currently export a `requireApproved` helper — add one there for consistency with other apps in the portfolio.

### M-2 · No rate limiting on expensive third-party routes
**File:** `src/app/api/practice/generate-station/route.ts`, `src/app/api/spotify/sync-playlist/route.ts`  
**Finding:** Both routes make outbound calls (Anthropic, Spotify) without any per-user throttle. A single client can drain the Anthropic/Spotify token budget through rapid automated calls.  
**Fix:** Apply the in-memory rate limiter pattern from the Scheduler (`lib/rate-limit.ts`) or add a per-user last-called column to enforce a cooldown (e.g., 10 s between `/generate-station` calls).

### M-3 · `logoDataUrl` stored without size cap
**File:** `src/app/api/roster/route.ts` (PUT handler)  
**Finding:** The roster PUT route writes `logoDataUrl` directly from the request body with no size check. The Onboarding UI caps file upload at 2 MB, but direct API callers bypass this, allowing arbitrarily large base64 payloads to be written to Postgres.  
**Fix:** Add a server-side check: reject `logoDataUrl` values longer than ~2.7 MB (2 MB binary → ~2.7 MB base64) with HTTP 413.

### M-4 · `constraint_overrides` table is dead code
**File:** `src/lib/schema.ts`, `drizzle/` migrations  
**Finding:** The table exists in schema and migrations but is not read or written by any route, component, or utility. It adds noise to the DB layer and future migrators.  
**Fix:** Drop the table in a new migration (`drizzle-kit generate` → apply). Remove from `schema.ts`.

## Low

### L-1 · `practice-pdf.ts` duplicates `loadPennant`
**File:** `src/lib/practice-pdf.ts`, `src/lib/pdf.ts`  
**Finding:** `practice-pdf.ts` contains its own `loadPennant` implementation with its own local cache, duplicating the version exported from `pdf.ts`. Two cache entries means the logo image can be in memory twice; divergence is a future maintenance risk.  
**Fix:** Import `loadPennant` from `pdf.ts` in `practice-pdf.ts` and delete the local copy.

### L-2 · No Content-Security-Policy header
**File:** `next.config.ts`  
**Finding:** Security headers include `X-Frame-Options`, `X-Content-Type-Options`, and `Referrer-Policy` but no CSP. Without a CSP, injected scripts (e.g., via a future XSS) have no browser-side restriction.  
**Fix:** Add a CSP header in `next.config.ts`. Start with a permissive policy (`default-src 'self'`; allowlist Spotify domains for OAuth redirects) and tighten over time.

### L-3 · `storage.ts` localStorage helpers are stale
**File:** `src/lib/storage.ts`  
**Finding:** `loadRoster` / `saveRoster` are no longer called from production code. `addHistoryEntry` and `clearAbsences` are still imported as pure logic helpers but carry dead localStorage code paths.  
**Fix:** Extract `addHistoryEntry` and `clearAbsences` as pure functions into a separate utility file. Delete `storage.ts`.

### L-4 · No test suite
**Finding:** Zero automated tests. The backtracking scheduler (`lib/scheduler.ts`) and constraint validation are pure functions well-suited to deterministic unit tests.  
**Fix:** Add `vitest` and cover at minimum `generateGameSheet` with standard 6-inning / 10-player inputs and known constraint violations.
