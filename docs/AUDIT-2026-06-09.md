## High

### H1 — Admin GET leaks Spotify OAuth tokens
**File:** `src/app/api/admin/users/route.ts:21`

`db.select().from(users)` returns **all columns** including `spotifyAccessToken` and `spotifyRefreshToken`. These are valid OAuth credentials that grant access to the linked Spotify account. Any admin (or any attacker who obtains an admin session) can read all coaches' Spotify tokens from the API response.

**Fix:** Project only non-sensitive columns in the select. Avoid `select *` on the users table in the admin list endpoint.

**Status: FIXED** — see commit in `audit/2026-06-09` branch.

---

### H2 — `history` POST writes unvalidated user input to database
**File:** `src/app/api/history/route.ts:41-47`

The POST handler extracts `date` and `players` from the request body and inserts them directly into `game_history` with no type checking, length limits, or sanitization:

```typescript
const { date, players } = body;
const [inserted] = await db
  .insert(gameHistory)
  .values({ userId, teamId: team.id, date, players })
  .returning();
```

A logged-in user can store arbitrary JSON of unlimited size in `game_history.players` and any string in `date`. This is a stored injection / data-integrity issue. Neon/Postgres will accept anything in a `json` column.

**Fix:** Validate `date` is a non-empty string (max ~50 chars) and `players` is an array with max 20 items, each item having the expected fields (`id: string`, `name: string`, `rank: number`, `absent: boolean`).

**Status: FIXED** — see commit in `audit/2026-06-09` branch.

### H3 — Admin PATCH does not validate email format on update
**File:** `src/app/api/admin/users/route.ts:113`

The POST endpoint validates email format with a regex, but the PATCH handler skips that check:

```typescript
if (email !== undefined) updates.email = email.toLowerCase().trim();
```

An admin can set any user's email to an invalid/malformed string (e.g., an empty string or a very long value), which could break downstream email delivery and uniqueness checks.

**Fix:** Apply the same email regex and length check in the PATCH path as in POST.

**Status: NOT FIXED** (low-risk in practice because only admins can call PATCH, but worth doing for correctness).

## Medium

### M1 — Debug console.log in production scheduler
**File:** `src/lib/scheduler.ts:421-424`

Four `console.log` calls fire on every roster generation:

```typescript
console.log(`[SOLVER] Inning 1 player order: ${ordered.map(p => `${p.name}(r${p.rank}...`).join(', ')}`);
console.log(`[SOLVER] premiumIF order: ...`);
console.log(`[SOLVER] ofFirst order: ...`);
console.log(`[SOLVER] topThreshold: ${topThreshold}`);
```

These run client-side in the browser (the scheduler is a client-side module), so they appear in every user's browser console. They expose player names and internal scoring details which is undesirable from a privacy perspective. They also add non-trivial string-building overhead on every generate call.

**Fix:** Remove the four `console.log` calls.

**Status: FIXED** — see commit in `audit/2026-06-09` branch.

### M2 — `logoDataUrl` has no size limit in roster PUT
**File:** `src/app/api/roster/route.ts:112-113`

The roster PUT accepts `config.logoDataUrl` and stores it in the `teams.logo_data_url` column without a size check:

```typescript
if (config?.logoDataUrl !== undefined) {
  teamUpdates.logoDataUrl = config.logoDataUrl;
}
```

A base64-encoded PNG can be megabytes. Storing large payloads in every DB row for every team inflates the database, can hit Neon row/transaction size limits, and slows all team reads.

**Fix:** Cap `logoDataUrl` at ~300 KB (roughly 400 KB as base64, which is plenty for a thumbnail). Reject or strip if larger.

**Status: NOT FIXED** — requires product decision on acceptable size.

### M3 — No rate limiting on AI/Spotify endpoints
**Files:** `src/app/api/practice/generate-station/route.ts`, `src/app/api/spotify/sync-playlist/route.ts`

Both endpoints make expensive third-party API calls (Anthropic, Spotify) with no throttling. Any approved user can hammer them:
- `/api/practice/generate-station`: each call costs real Anthropic tokens.
- `/api/spotify/sync-playlist`: each call makes multiple Spotify API calls and mutates data (creates/updates a playlist).

**Fix:** Add simple per-user rate limiting (e.g., a KV-backed counter or Vercel's built-in rate limiting). For the Spotify sync, consider at minimum a debounce window (e.g., 60-second cooldown per team).

**Status: NOT FIXED** — requires infrastructure decision.

### M4 — Fragile player ID generation
**File:** `src/app/page.tsx:177-178`

```typescript
const maxId = players.length > 0 ? Math.max(...players.map((p) => parseInt(p.id))) : 0;
const newId = String(maxId + 1);
```

`parseInt` on a non-numeric player ID (e.g., a UUID from a future migration or import) returns `NaN`. `Math.max(...[NaN])` returns `NaN`. `NaN + 1` is `NaN`. All new players would get id `"NaN"`, breaking the roster.

**Fix:** Use `crypto.randomUUID()` for new player IDs (already available in modern browsers and Edge runtime). This is a pre-existing fragility that should be addressed before any roster import or ID migration.

**Status: FIXED** — see commit in `audit/2026-06-09` branch.

### M5 — `practice-pdf.ts` duplicates `loadPennant` with its own cache
**File:** `src/lib/practice-pdf.ts:8-27`

`practice-pdf.ts` defines its own local `loadPennant()` function with its own `pennantCache` variable, instead of importing the exported `loadPennant` from `pdf.ts`. This means:
1. Two separate fetches of `/logo.png` may occur if both PDF types are generated in the same session.
2. Cache state is duplicated.

**Fix:** Remove the local copy and import `loadPennant` from `./pdf`.

**Status: NOT FIXED** — minor correctness/duplication issue, low risk.

### L3 — `drizzle-kit` in `dependencies` instead of `devDependencies`
**File:** `package.json:18`

`drizzle-kit` is a migration CLI tool that is never imported at runtime. It should be in `devDependencies` to avoid being bundled or installed in production environments.

**Fix:** Move `drizzle-kit` to `devDependencies`.

**Status: FIXED** — see commit in `audit/2026-06-09` branch.

### L4 — No Content-Security-Policy header
**File:** `next.config.ts`

The app sets `X-Frame-Options`, `X-Content-Type-Options`, and `Referrer-Policy`, but no `Content-Security-Policy`. A CSP would add meaningful defense-in-depth against XSS, particularly given the app handles OAuth redirects, user-provided team names, and renders player names in the DOM.

**Fix:** Add a `Content-Security-Policy` header in `next.config.ts`. Start with a report-only policy to identify violations before enforcing.

**Status: NOT FIXED** — requires careful crafting to avoid breaking inline styles, Next.js scripts, and third-party fonts/images.

## Notes

- **No hardcoded secrets found.** All credentials are read from `process.env`.
- **CSRF:** The session cookie uses `sameSite: "lax"` which provides standard CSRF protection for state-changing requests via form submissions and cross-origin navigations. API routes that are called with `fetch` + JSON body are safe by default.
- **SQL injection:** All DB queries use Drizzle ORM's parameterized queries. No raw string interpolation in SQL was found.
- **OAuth state forgery:** Both Google and Spotify flows use HMAC-signed state with a nonce cookie — CSRF on the OAuth flow is well-mitigated.
- **Timing attacks:** HMAC comparisons in `session.ts` use `crypto.timingSafeEqual`; middleware uses a manual constant-time loop (required for Edge runtime). Both are correct.
