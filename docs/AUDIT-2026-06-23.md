## Medium

### M3 — Admin PATCH does not validate email format on update *(carry-over)*
**File:** `src/app/api/admin/users/route.ts` — previously at line 129, now ~131 after fix

The `POST` endpoint for creating a user validates email format with a regex, but the `PATCH` handler previously skipped that check:

```typescript
// before fix:
if (email !== undefined) updates.email = email.toLowerCase().trim();
```

An admin could update any user's email to an invalid or empty string, breaking downstream email delivery and potentially violating database uniqueness constraints in confusing ways.

**Fix applied:** Email validation added to PATCH path, matching the POST logic:

```typescript
if (email !== undefined) {
  const emailRegex = /^[^\s@]+@[^\s@]+\.[^\s@]+$/;
  const trimmed = typeof email === "string" ? email.toLowerCase().trim() : "";
  if (!trimmed || !emailRegex.test(trimmed)) {
    return NextResponse.json({ error: "Invalid email format" }, { status: 400 });
  }
  updates.email = trimmed;
}
```

**Status: FIXED** — see commit in this audit.

### M2 — `logoDataUrl` has no size limit in roster PUT *(carry-over)*
**File:** `src/app/api/roster/route.ts:112-113`

The roster `PUT` handler accepts `config.logoDataUrl` and stores it in `teams.logo_data_url` without a size check. A base64-encoded PNG can be several megabytes. The Onboarding UI client-side caps logo file uploads at 2 MB, but this is bypassable with a direct API call. Large values inflate every DB row for the team, can hit Neon row/transaction size limits, and slow all team reads that return the config blob.

**Recommended fix:** Cap `logoDataUrl` at ~400 KB (base64) at the API layer:

```typescript
if (config?.logoDataUrl !== undefined) {
  if (typeof config.logoDataUrl === "string" && config.logoDataUrl.length > 400_000) {
    return NextResponse.json({ error: "Logo image too large (max ~300 KB)" }, { status: 413 });
  }
  teamUpdates.logoDataUrl = config.logoDataUrl;
}
```

**Status: NOT FIXED** — requires product decision on acceptable size threshold.

### M4 — No rate limiting on AI and Spotify endpoints *(carry-over)*
**Files:** `src/app/api/practice/generate-station/route.ts`, `src/app/api/spotify/sync-playlist/route.ts`

Both endpoints make expensive third-party API calls with no throttling. Any approved user can hammer them:
- `/api/practice/generate-station`: each call consumes Anthropic tokens (Haiku 4.5, `max_tokens: 600`).
- `/api/spotify/sync-playlist`: each call makes multiple Spotify API calls (probe + create/update + replace tracks) and can be triggered repeatedly by clicking Generate.

**Recommended fix:** Add per-user rate limiting via a lightweight KV counter (Vercel KV / Upstash Redis). For the Spotify sync specifically, a server-side 60-second cooldown per team would prevent most abuse with minimal complexity.

**Status: NOT FIXED** — requires infrastructure decision.

## Low

### L1 — No Content-Security-Policy header *(carry-over)*
**File:** `next.config.ts`

The app sets `X-Frame-Options`, `X-Content-Type-Options`, and `Referrer-Policy` but no `Content-Security-Policy`. A CSP would provide meaningful defense-in-depth against XSS, particularly relevant for the OAuth redirect flows and admin HTML response pages.

**Recommended fix:** Add a `Content-Security-Policy` header in `next.config.ts`. Start with `Content-Security-Policy-Report-Only` to identify violations before enforcing. A restrictive policy for this app would allow `self` for scripts/styles and `open.spotify.com` / `i.scdn.co` for images.

**Status: NOT FIXED** — requires careful crafting to avoid breaking inline styles, Next.js scripts, and Spotify image URLs.

### L2 — `practice-pdf.ts` duplicates `loadPennant` with its own cache *(carry-over)*
**File:** `src/lib/practice-pdf.ts:8-27`

`practice-pdf.ts` defines its own local `loadPennant()` function with its own `pennantCache` variable, rather than importing the exported version from `pdf.ts`. This means two separate fetches of `/logo.png` may occur if both PDF types are generated in the same session, and cache invalidation is split across two module-scoped variables.

**Recommended fix:** Remove the local copy and `import { loadPennant } from "./pdf"`.

**Status: NOT FIXED** — minor duplication, low risk.

### L3 — `constraint_overrides` table is dead code *(carry-over)*
**File:** `src/lib/schema.ts:77-83`

The `constraintOverrides` table exists in the schema and migrations but is never read or written by any route or UI component. Constraint state lives in `teams.constraintConfig` JSON.

**Recommended fix:** Remove via a `DROP TABLE constraint_overrides` migration and remove the `constraintOverrides` export from `schema.ts`.
